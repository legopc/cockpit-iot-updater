# Troubleshooting Guide

---

## Sidecar not reachable

**Symptom:** Cockpit page shows "Could not reach sidecar (is iot-updater.service running?)"

```bash
# Check service status
systemctl status iot-updater.service

# Check it's listening
curl http://127.0.0.1:8088/status

# View logs
journalctl -u iot-updater.service -n 50
```

**Common causes:**
- Service not started: `systemctl enable --now iot-updater.service`
- Python syntax error: `python3 /var/lib/iot-updater/server.py` (run manually to see error)
- Port conflict: `ss -tlnp | grep 8088`

## "IoT Updater" not in Cockpit sidebar

**Symptom:** The page doesn't appear in Cockpit's navigation

```bash
# Check the page files exist
ls /usr/share/cockpit/iot-updater/
# Should show: manifest.json  index.html  update.js

# Restart Cockpit to reload plugins
systemctl restart cockpit.socket

# Check Cockpit version (need 232+)
cockpit-bridge --version
```

If Fedora IoT's `/usr` is read-only and the files aren't persisting:
```bash
# Check if it's actually an overlay issue
ls -la /usr/share/cockpit/iot-updater/

# Files in /usr/share/ ARE writable on Fedora IoT (OSTree allows overlay writes)
# but they'll be gone after the next OS update — install via RPM for permanence
```

## Upload fails or stalls

**Symptom:** Progress bar stops, browser shows network error

```bash
# Check sidecar is still alive during upload
journalctl -u iot-updater.service -f

# Check disk space (must have 2× bundle size free in /var/tmp)
df -h /var/tmp

# Check partial file
ls -lh /var/tmp/iot43-update.iotupdate
```

**Common causes:**
- `/var/tmp` full: `rm /var/tmp/iot43-update.iotupdate` then free space
- Browser timeout: try a different browser; avoid Safari for large uploads
- Connection reset: ensure your LAN is stable; Gigabit recommended for 2 GB files

## "Bundle error: version.json missing or invalid"

**Symptom:** After upload, error appears instead of version preview

The `.iotupdate` file is either:
- Not generated by `make-bundle.sh` (wrong format)
- Corrupted during transfer

```bash
# Verify the bundle on the source machine before uploading
tar -tvf /path/to/bundle.iotupdate   # should list version.json and update.delta
tar -xOf /path/to/bundle.iotupdate version.json   # should print valid JSON
```

## `ostree static-delta apply-offline` fails

**Symptom:** Apply starts but fails with an OSTree error in the logs

```bash
journalctl -u iot-update.service -n 100
```

**Common error messages:**

| Error | Cause | Fix |
|-------|-------|-----|
| `Delta checksum mismatch` | Bundle corrupted during upload | Re-upload the bundle |
| `No commit found for from commit` | `from_commit` not in local repo | Generate bundle with correct `--from` hash |
| `from_commit mismatch` | Device was updated/rolled back since bundle was made | Regenerate bundle with new `--from` |
| `No space left on device` | `/ostree/repo` or `/` full | Free space; OSTree repo at `/ostree/` needs ~2× delta size |

## `rpm-ostree deploy` fails

**Symptom:** OSTree apply succeeds but rpm-ostree step fails

```bash
# Run manually to see full error
rpm-ostree deploy :<commit-hash>

# Check rpm-ostree status
rpm-ostree status
```

**Common causes:**
- Layered packages conflict with the new commit: `rpm-ostree override remove <pkg>`
- Pending undeployed commit already exists: `rpm-ostree cleanup -p`

---

## Device reboots but boots old version

**Symptom:** Device reboots, Cockpit page reconnects, `rpm-ostree status` shows old version

```bash
# Check pending deployments
rpm-ostree status

# Check OSTree admin
ostree admin status
```

This usually means `rpm-ostree deploy` reported success but the new deployment wasn't
set as default. Try:
```bash
rpm-ostree deploy :<commit-hash>
systemctl reboot
```

If it still boots the old version, the GRUB bootloader entry may not have been updated.
Check `/boot/loader/entries/` for the expected entry.

## History file corrupted

```bash
# Reset history (loses audit trail, but sidecar recovers)
echo '[]' > /var/lib/iot-updater/history.json

# Or view the raw file
cat /var/lib/iot-updater/history.json | python3 -m json.tool
```

## Manual update (bypassing the UI)

If the Cockpit page is unavailable, apply the update manually:

```bash
# Copy bundle to device
scp bundle.iotupdate user@device:/var/tmp/iot43-update.iotupdate

# On the device as root:
sudo /var/lib/iot-updater/apply-update.sh

# The script will:
# 1. Extract and validate the bundle
# 2. Apply the OSTree delta
# 3. Deploy via rpm-ostree
# 4. Reboot after 5 seconds
```

---

## Rollback after failed update

```bash
# On the device as root
rpm-ostree rollback
systemctl reboot
```

OSTree always keeps the previous deployment. After rollback, update the history entry manually:
```bash
python3 -c "
import json
h = json.load(open('/var/lib/iot-updater/history.json'))
if h: h[-1]['status'] = 'rolledback'
json.dump(h, open('/var/lib/iot-updater/history.json','w'), indent=2)
"
```
